# Agent UI Plan

## Статус

Этот файл — рабочий план и фиксация текущего состояния папки `agent_ui`.

На текущем этапе собран отдельный локальный UI для табличных case и будущего общего runtime для UI/TG.

Архитектурно `agent_ui` теперь нужно читать вместе с:

- `../docs/core/target-runtime-structure.md`
- `../docs/core/connectors-ui-plan.md`

## Что уже сделано

### 1. Отдельное приложение

Собрана отдельная папка `agent_ui`, чтобы не смешивать интерфейс и оркестрацию с кодом модулей.

Текущий состав:

- `backend/` — FastAPI backend;
- `frontend/` — HTML/CSS/JS интерфейс;
- `.env` и `.env.example` — локальная конфигурация;
- `README.md` в корне и в подпапках.

### 2. Backend-слой

Уже собраны и подключены:

- контейнер приложения;
- runtime-state;
- job-store;
- intake/workbook для inspect и создания case-root;
- workspace-сервис для работы с case-root и текущим case;
- OCR-слой с записью в `work/ocr.json`;
- reporting/ui для сборки экранного ответа;
- отдельный `eco_fee`-сервис для книги `Расчет экосбора.xlsx`;
- chat/core + chat/scenarios для case-aware чата;
- API-маршруты для:
  - `workbook`
  - `jobs`
  - `workspace`
  - `chat-cli`
  - `eco-fee`

### 3. Провайдеры и конфигурация

Сделана конфигурация через `.env`.

AI-слой собран через `integrations/ai`.

- активный provider и runtime-настройки читаются через backend-конфигурацию;
- UI не должен жить отдельным экраном выбора runtime-конфигурации;
- логика вызова моделей не размазывается по `main`, `workspace` и transport-слоям.

### 4. Работа с кейсами

Собран workspace-слой поверх `storage/cases`.

UI и backend уже умеют:

- подцеплять активный `case-root` после workbook intake;
- выбирать активный root;
- загружать текущий кейс;
- читать:
  - `case.json`
  - `source_row.json`
  - `images/*`
  - `work/status.json`
  - `work/ocr.json`
  - `result/ui_response.json`

### 5. Новый экран review

Текущий экран переделан под компактный операторский формат.

Сейчас он состоит из:

- верхней панели:
  - `Загрузить таблицу`
  - выбор листа и диапазона строк
  - флаги автозапуска и проверок
  - `Создать товары`
- рабочей панели:
  - `Обновить экран`
  - `Подобрать текущий`
  - `Подобрать 5 подряд`
  - `Остановить подбор`
  - счетчики и batch-статусы
  - `Дальше`
  - `Записать в Excel`
- верхней карусели строк;
- левой колонки:
  - изображения товара
  - входного анализа по товару
  - краткого OCR для оператора
  - подробного OCR для проверки
- центральной колонки:
  - case-aware чат и контекст
- правой колонки:
  - `ТНВЭД`
  - критерии и обоснование
  - альтернативы
  - `IFCG / Sigma / ITS / Экосбор`
  - будущие расчеты и writeback-поля

Текущая договоренность по UI:

- UI не должен сам собирать экран из сырых `work/*.json`;
- backend должен отдавать уже собранный экранный объект через `reporting/ui`;
- OCR-панель теперь разделена на короткий операторский слой и подробный слой для проверки;
- batch OCR/GPT должен восприниматься как backend job, а не как frontend-магия.

### 6. Логика статусов

Уже реализовано:

- выбор кейса через карусель;
- подсветка текущего кейса;
- зеленый статус после `Записать в Excel`;
- красный статус после `Дальше`;
- OCR/GPT статусы по кейсу и в карусели;
- пакетный запуск по `5` кейсам;
- batch-баннер с прогрессом;
- остановка batch/job с фронта через backend stop-логику.

### 7. Работа с workbook intake

Сделан отдельный workflow загрузки таблицы.

Он уже умеет:

- принимать файл Excel или путь;
- делать `inspect`;
- выбирать лист;
- задавать диапазон строк;
- создавать `case-root` как background job;
- автоматически подцеплять новую папку кейсов;
- опционально запускать первый OCR batch после создания cases.

Старый donor-script больше не должен быть центром нового runtime: workbook intake живет внутри backend.

### 8. Чат по кейсу

Центральный чат больше не заглушка.

Теперь он работает через `chat/core` и UI-сценарий case-чата:

- backend собирает `context packet` по текущему кейсу;
- пишет его в `work/06_chat_cli_context.md`;
- хранит историю в `work/06_chat_cli.json`;
- вызывает `codex exec` внутри папки кейса;
- ответ возвращается прямо в центральное окно `agent_ui`.

Что это дает уже сейчас:

- чат живет отдельно по каждому товару;
- не тратит деньги, пока оператор молчит;
- может использовать локальный контекст кейса и внешний поиск;
- не смешивает товары между собой.

Текущий runtime-факт:

- `history` и `message` уже живые;
- проверка через `TestClient` и браузер показала, что `Codex CLI` отвечает по реальному кейсу и пишет артефакты в `work/`.
- быстрые подсказки модели в центре переведены в компактную тонкую полосу с раскрытием по кнопке;
- отправка чата теперь работает по `Enter`, а `Shift+Enter` оставляет перенос строки;
- нижний поясняющий текст под чатом убран;
- добавлен режим `Фокус`, который скрывает левую колонку и расширяет рабочую область под чат и правый вывод.
- основная рабочая сетка переведена в более жесткий desktop-grid: три колонки тянутся до нижней границы окна, а прокрутка живет внутри самих панелей.

Пока заглушкой остается только запись результата обратно в Excel.

## Следующий слой интеграции

Новая договоренность по migration order:

- из `2. ВЭД просто` в `agent_ui/backend` тянем прежде всего table/writeback слой;
- из `3. БОТ` в `agent_ui/backend` тянем прежде всего правую предметную логику:
  - `Sigma`
  - `ИТС`
  - `IFCG`
  - `PP1637`
  - report rhythm

То есть UI дальше не развивается как самостоятельная абстрактная витрина.
Он становится окном для живых stage-модулей pipeline, где:

- таблица и запись в Excel приходят из table donor-слоя;
- правая колонка и review-pack приходят из donor-логики `БОТа`;
- case-chat остается боковым режимом для сложных вопросов.

Новая договоренность:

- полноценный модуль экосбора проектируем отдельно в `agent_system/03_costs/01_eco_fee/`;
- в `v1` это должен быть lookup по коду + ручной выбор eco-группы + калькулятор;
- правая часть UI должна ориентироваться не на абстрактные карточки, а на фактическую структуру ответа из Telegram:
  - статусы источников;
  - основной код;
  - строка `ИТС | ПОШ | ЭКО | СТП`;
  - описание;
  - критерии;
  - обоснование;
  - альтернативы;
  - статус;
  - и ниже секция `Экосбор` по текущему коду.

Что уже сделано по runtime экосбора:

- экосбор разделен на два слоя:
  - `backend/app/storage/knowledge/catalogs/eco_fee`
  - `backend/app/calculations/eco_fee`
- книга `Расчет экосбора.xlsx` теперь читается в отдельный каталог, а не в snapshot-заглушку;
- доступны реальные backend-маршруты:
  - `GET /api/eco-fee/reference`
  - `POST /api/eco-fee/lookup`
  - `POST /api/eco-fee/calculate`
  - `GET /api/eco-fee/currency-rates`
- lookup умеет:
  - нормализовать код;
  - искать exact/prefix/expanded совпадения;
  - группировать их по eco-группам;
  - отдавать оператору реальные названия из базы;
- calculate умеет:
  - считать товарную часть;
  - считать упаковочную часть;
  - считать итог `руб / USD / USD на кг`.

Текущая договоренность по UI-экосбору уже другая:

- основной экран читает экосбор из `current_case.eco_fee` в общем `/api/workspace`;
- отдельные `lookup/calculate` остаются как reference/debug endpoints, но не как основной UI-flow;
- UI показывает:
  - все наименования из БД по совпадению;
  - сноски `<N>`;
  - top-10 raw-строк из БД;
  - модалку проверки конкретной строки;
  - helper-подсказки `USD / EUR` по ЦБ РФ.

Последняя правка по правой колонке:

- карточка `СТП` теперь автоматически превращается в `СТП + ЭКО`, если для выбранного кода есть eco-доплата;
- под карточками сохраняется формула в явном виде:
  - `СТП + ЭКО`

- нижние большие `details`-секции убраны;
- нижний support-dock больше не дублирует `Sigma / ITS / Customs / Экосбор`;
- `IFCG` сейчас уже привязан к выбранной альтернативе и показывает компактную review-сводку:
  - `что проверяли`
  - `лучшие коды по практике`
  - `почему такой вывод`.

Важная оговорка по архитектуре:

- текущий `UI` уже тянет живую pipeline-логику по основному workspace-path;
- реально сейчас он завязан на:
  - `intake/workbook`;
  - `workspace` поверх `storage/cases`;
  - `orchestrator/pipelines/case_pipeline.py`;
  - `reporting/ui/workspace`;
  - `storage/knowledge/catalogs/eco_fee + calculations/eco_fee`.
- placeholder-ветка остается только fallback-режимом для legacy-кейсов без нового pipeline-result.

Что уже сделано по правой колонке:

- правая часть перестроена под формат, близкий к реальному ответу из Telegram;
- теперь она рендерит:
  - `РО / ИТС БОТ / SIGMA-SOFT`
  - `Текст`
  - основной `ТНВЭД` со ссылками на `Sigma` и `IFCG`
  - строку `ИТС | ПОШ | ЭКО | СТП`
  - `Описание ТНВЭД`
  - `Критерии выбора`
  - `Обоснование`
  - `Альтернативы`
  - `Статус`
  - ниже — секцию `Экосбор`
- альтернативный код можно сделать основным кликом по нему;
- `Экосбор` теперь привязан к выбранному коду, а не висит отдельной абстракцией;
- если eco-совпадений нет, показывается честный `not_found` без калькулятора;
- если eco-совпадения есть, доступны:
  - явный вывод `Наименования в базе экосбора`;
  - показ кодов, по которым найдено совпадение;
  - показ сносок `<N>`;
  - top-10 строк БД для operator review;
  - расчет доплаты `ЭКО к СТП` на `1 кг`;
  - строка сравнения `СТП + ЭКО = итог`.

Последняя фиксация по экосбору:

- для операторского review `v1` экосбор считаем не как общий калькулятор по массе, а как доплату к `СТП` на `1 кг`;
- если по коду найден лучший match, справа показывается:
  - `ЭКО к СТП`
  - `СТП + ЭКО`
  - наименование группы из базы
  - примеры наименований из книги экосбора;
- если по коду экосбора нет, интерфейс честно пишет:
  - `по коду экосбор не найден`;
- если по коду найдено несколько eco-групп, интерфейс показывает полную сводку по совпадениям и raw-строкам БД, чтобы оператор сам проверил релевантную группу.

Последние фронтовые доработки:

- левая колонка с фото стала адаптивной по числу изображений:
  - `1` фото — одно крупное полотно;
  - `2` фото — две полноценные плитки на всю ширину блока;
  - `3+` фото — компактная сетка без пустого провала под картинками;
- из-за этого убран визуальный баг, когда панель фото была высокой, а сами изображения жили только в верхней полосе;
- для кейсов после intake/workbook, где кодов еще нет, правый блок больше не делает вид, что уже “тянет eco-базу”;
- в пустом состоянии теперь честно пишется:
  - `код пока не выбран`
  - `Экосбор появится после того, как по товару будет выбран или предложен код`.
- блок `ТНВЭД` справа ужат в более плотный формат:
  - строка `ИТС / ПОШ / НДС / СТП(+ЭКО)` теперь встроена прямо внутрь секции кода;
  - отдельный блок `Статус` убран полностью;
  - смайлики `🧾 🛡️ 💰 🏷️ ♻️` больше не рисуются “по умолчанию”, а показываются только если по коду есть реальный сигнал;
  - `Маркировка / ЧЗ` переименована в `Честный знак`;
  - для `СТП + ЭКО` формула теперь сидит прямо внутри компактной ячейки, без дубля отдельными строками.
- следующий шаг по этой же зоне уже частично сделан:
  - сам код `ТНВЭД` вынесен в чистую верхнюю строку;
  - ниже появился общий “поддон” со стикерами ставок и маркеров;
  - это нужно, чтобы 10-значный код и длинная полоса `🧾 🛡️ 💰 🏷️ ♻️` не спорили за одно место в строке;
  - `Эко-сигнал` сокращён до `Эко`.

Также добавлены отдельные backend-сервисы:

- `source_workbook_service.py`
- `analysis_placeholder_service.py`

Они нужны, чтобы UI не вытаскивал Excel и длинный анализ напрямую из `workspace_service`.

## Текущий сценарий работы

Рабочий поток сейчас такой:

1. открыть `agent_ui`;
2. загрузить таблицу через workbook drawer;
3. выбрать лист и диапазон строк;
4. создать cases и при необходимости запустить стартовый OCR batch;
5. открыть нужный кейс из верхней карусели;
6. смотреть вход, OCR и правую сводку;
7. при необходимости запускать `Подобрать текущий` или `Подобрать 5 подряд`;
8. либо нажать `Записать в Excel`, либо `Дальше`.

## Что дальше

Следующие этапы:

1. Переименовать batch-layer из старого `batch_ocr / dispatch_ocr_batch` в нейтральный pipeline naming.
2. Реально писать результат обратно в Excel.
3. Довести TG transport до того же channel-neutral pipeline/result path, что и UI.
4. Сжать remaining placeholder/fallback-хвосты только до legacy-case режима.
5. Доделать экосбор до полноценного сценария с несколькими упаковочными строками.
6. Подключить тот же backend worker/runtime к Telegram-боту.
7. Добавить более явную управляемую параллельность и naming для batch-задач через orchestrator.
8. Добавить для case-чата быстрый и глубокий режимы без разрастания transport-логики.
